# 研究执行引擎（Research Engine）

> **定位**：Stage 4 研究执行的搜索操作手册
>
> **核心模式**：A+B 并行（Subagent）+ D→E→F→G→C 串行（主 Session）
>
> **关联模块**：
> - [`data_sources.md`](./data_sources.md) - 必选数据源清单
> - [`triangulation.md`](../methodology/triangulation.md) - 三角验证法
> - [`hypothesis_driven.md`](../methodology/hypothesis_driven.md) - 假设驱动法
> - [`interview.md`](../methodology/interview.md) - 专家访谈法

---

## 核心原则

1. **假设驱动**：每次搜索都服务于验证/证伪某个假设
2. **数据源优先**：必须优先使用 `data_sources.md` 中的 P0-P2 级数据源
3. **三角验证**：核心数据必须寻找 2-3 个独立来源
4. **证据链可追溯**：每个结论都能追溯到原始来源
5. **执行顺序**：A+B 并行 → D → E → F → G → C，依次执行
6. **信息透明**：每个轨道执行前向用户播报，跳过时告知原因

### 轨道定义

| 轨道 | 名称 | 执行方式 | 数据源 |
|------|------|---------|--------|
| **A** | 公开数据收集 | Subagent 并行 | GoogleSearch + 网页抓取 |
| **B** | 数据源定向搜索 | Subagent 并行 | 权威数据源官网（艾瑞、头豹等） |
| **D** | 语雀文档库 | 主 Session | 内部历史研究、笔记 |
| **E** | 小红书 | 主 Session | 消费者舆情、产品反馈 |
| **F** | ODPS 内部数据 | 主 Session | 业务数据、用户行为 |
| **G** | 用户原声数据 | 主 Session | 用户体验问题、投诉痛点、满意度 |
| **C** | 专家访谈 | 主 Session | 访谈纪要（需用户配合） |

## 执行规则

### Track A+B：并行执行（Subagent）

**触发条件**：所有研究项目必选

**执行方式**：
1. 主 Session 分解任务，生成分发清单
2. 启动 2 个 Subagent 并行执行
3. 等待结果返回，汇总证据

**进度播报**：
```
━━━ Track A+B 并行执行 ━━━
📚 启动 Subagent 1：公开数据收集
📚 启动 Subagent 2：数据源定向搜索
⏳ 等待结果返回...

✅ Track A 完成：找到 X 条证据
✅ Track B 完成：找到 X 条证据
```

---

### Track D-G：依次执行（主 Session）

**执行原则**：
- 原则上都执行
- 判断无意义则跳过，但必须告知用户原因
- 无权限/无数据时告知原因 + 提醒配置 + 跳过

**跳过判断标准**：

| 轨道 | 跳过条件 | 示例 |
|------|---------|------|
| **D 语雀** | 无权限 / 无相关文档 | "未检测到语雀账号权限，跳过此轨道" |
| **E 小红书** | 非消费类议题 / 无权限 | "议题为 B 端业务，小红书数据价值较低，跳过此轨道" |
| **F ODPS** | 无权限 / （外部研究 **且** 用户非行业参与者） | "本次为纯外部行业研究且无内部业务数据，跳过此轨道" |
| **G 用户原声** | 无权限 / （非产品相关议题 **且** 用户非行业参与者） | "本次为纯外部行业研究且无用户反馈数据，跳过此轨道" |

> **行业参与者判定**：用户在 Stage 1 `user_brief.md` 中表明自身是研究行业的现有企业/从业者（如"我们是猫粮公司，研究猫粮市场"），则视为行业参与者。此时 D/F/G 应默认激活探查，因为内部数据可能包含市场洞察（如用户画像、竞品对比、销售趋势等）。

**进度播报**：
```
━━━ Track D：语雀文档库 ━━━
📚 加载工具：skylark_search
🔍 搜索关键词：[关键词]
✅ 找到 X 篇相关文档

━━━ Track E：小红书 ━━━
📚 加载工具：scripts/xhs/ (TikHub API)
🔍 搜索关键词：[关键词]
⚠️ 跳过此轨道：议题为 B 端 SaaS 业务，小红书数据价值较低

━━━ Track F：ODPS 内部数据 ━━━
📚 加载工具：search_tables → execute_sql
⚠️ 跳过此轨道：未检测到 ODPS 访问权限。如需接入请联系管理员配置

━━━ Track G：用户原声数据 ━━━
📚 加载工具：execute_sql（ODPS）
🔍 查询表：antcc.adm_crm_digital_life_wanqiu_all_detail_gov_di
✅ 找到 X 条相关问题记录
```

### Track C：专家访谈（Stage 3.5 工作流激活）

**触发条件**：
- 关键数据无法从公开渠道获取
- 需要验证假设但公开信息不足
- 用户明确有专家资源可用

**执行方式**：
1. 基于 A+B+D+E+F+G 的初步发现设计访谈提纲
2. 与用户确认访谈对象和方式
3. 执行访谈或用户提供访谈记录
4. 整理访谈证据

**⛔ 访谈催收检查点**：

访谈是用户的异步动作，AI 不会自动等到结果。因此在 **Stage 4 所有其他轨道执行完毕后、生成 `evidence_base.md` 之前**，必须执行以下检查：

```
如果 Stage 3.5 曾被激活（interview_guides.md 已生成）：
    → 使用 AskUserQuestion 询问用户：
      "Stage 3.5 生成了访谈提纲，请问访谈进展如何？"
      选项：
      1. 已完成，我提供访谈记录/要点（→ 整理后纳入 evidence_base.md）
      2. 还在进行中，先用现有数据继续（→ 标注"访谈证据待补充"，Stage 7 可补入）
      3. 决定不做访谈了（→ 标注"访谈轨道取消"，继续）
```

无论用户选哪个，都在 `evidence_base.md` 的 Track C 部分记录状态。

## 异常处理

### 无权限/无数据

**处理流程**：
```
1. 检测到无权限或无数据
   ↓
2. 明确告知用户原因
   ↓
3. 提醒用户如何配置（如适用）
   ↓
4. 跳过此轨道，继续执行下一轨道
```

**播报模板**：
```
⚠️ Track [X] 跳过通知

原因：[具体原因]
- 无访问权限
- 议题类型不匹配
- 数据源暂不可用

建议：[配置建议]
- 如需接入请联系管理员
- 可手动提供数据

继续执行下一轨道...
```

---

### Subagent 超时

**处理**：
1. 等待其他 Subagent 完成
2. 对超时轨道使用已获取数据
3. 在证据库标注"数据不完整"

## 时间估算

| 阶段 | 轨道 | 预计时间 |
|------|------|---------|
| 并行 | A+B | 20-40 分钟 |
| 串行 | D | 5-10 分钟 |
| 串行 | E | 10-20 分钟 |
| 串行 | F | 10-20 分钟 |
| 串行 | G | 5-10 分钟 |
| 条件 | C | 30-60 分钟（如触发） |
| 整合 | 三角验证 | 15-20 分钟 |

**总计**：
- 不含访谈：65-120 分钟
- 含访谈：95-180 分钟

### 轨道激活决策矩阵

**三维度决策**：议题特性 × 数据源特性 × 用户权限

#### 维度一：议题特性 → 数据需求

| 议题类型 | 核心数据需求 | 必选轨道 | 可选轨道 | 用户为行业参与者时额外可选 |
|---------|-------------|---------|---------|--------------------------|
| 行业研究 | 市场规模、竞争格局、趋势 | A + B | D（历史研究） | **F**（内部业务数据佐证）、**G**（用户反馈印证市场需求） |
| 竞争分析 | 竞对数据、策略、动向 | A + B | C（深度信息）、D | **F**（内部竞品对比数据）、**G**（用户对竞品的反馈） |
| 消费者洞察 | 用户画像、需求、反馈 | A + B + E | D、G | **F**（用户行为数据） |
| 内部业务分析 | 业务数据、用户行为 | F + A + B | D（历史研究）、G | — |
| 商业机会挖掘 | 市场空白、未满足需求 | A + B + E | C、D | **F**（现有业务数据发现空白）、**G**（用户未满足需求） |
| 商业模式分析 | 盈利模式、单位经济 | A + B | C（竞对信息）、D | **F**（自身单位经济数据） |
| 新兴行业 | 行业定义、玩家、趋势 | A + B + C | D | — |

#### 维度二：数据源特性 → 适用场景

| 轨道 | 数据源特性 | 最适用场景 | 不适用场景 |
|------|-----------|-----------|-----------|
| **A 公开搜索** | 广泛、浅层、时效性强 | 所有议题的基础扫描 | 深度数据、内部数据 |
| **B 数据源定向** | 权威、结构化、有深度 | 需要权威数据支撑的议题 | 非常新兴、无权威报告的领域 |
| **C 专家访谈** | 深度、非公开、定制化 | 非公开信息、深度洞察验证 | 公开信息充足的议题 |
| **D 语雀文档库** | 内部沉淀、历史研究、方法论 | 有历史积累的话题 | 全新领域 |
| **E 小红书** | 消费者舆情、产品反馈、趋势 | 消费类、C端产品、品牌舆情 | B端业务、非消费领域 |
| **F ODPS内部** | 业务数据、用户行为、交易 | 内部分析、业务诊断、数据验证 | 外部研究且用户非行业参与者、无权限场景 |
| **G 用户原声** | 用户体验问题、投诉痛点 | 产品优化、用户体验分析、消费者洞察 | 纯外部行业研究且用户非行业参与者 |

#### 维度三：用户权限 → 轨道可用性

| 轨道 | 权限要求 | 无权限时处理 |
|------|---------|-------------|
| A / B | 无 | - |
| C | 需要专家资源或预算 | 跳过，仅用公开数据 |
| D | 语雀账号权限 | 跳过 |
| E | 小红书数据接口权限 | 跳过，或用户手动提供 |
| F | ODPS访问权限 | 跳过，或用户手动提供数据 |
| G | ODPS 访问权限（antcc_dev 项目） | 跳过，或用户手动提供数据 |

### 轨道激活流程（Layer 1 执行）

```
Step 1: 识别议题类型
    ↓
Step 2: 查询「议题特性→数据需求」表，确定必选/可选轨道
    ↓
Step 2.5: 检查用户身份——是否为研究行业的参与者（从 user_brief.md 判断）
          → 是：将「用户为行业参与者时额外可选」列的轨道加入可选清单
          → 否：维持原可选清单
    ↓
Step 3: 检查用户权限，标记可用/不可用轨道
    ↓
Step 4: 检查数据源特性适用性，调整轨道优先级
    ↓
Step 5: 输出「轨道激活清单」
```

**轨道激活清单模板**：

```markdown
## 轨道激活清单

### 议题类型
[行业研究 / 竞争分析 / 消费者洞察 / ...]

### 激活轨道
| 轨道 | 状态 | 原因 |
|------|------|------|
| A 公开搜索 | ✅ 激活 | 必选轨道 |
| B 数据源定向 | ✅ 激活 | 必选轨道 |
| C 专家访谈 | ⏸️ 待定 | 需用户确认是否有资源 |
| D 语雀文档库 | ✅ 激活 | 有历史研究可参考 |
| E 小红书 | ❌ 跳过 | 议题非消费类 |
| F ODPS内部 | ❌ 跳过 | 外部研究，无权限 |
| G 用户原声 | ❌ 跳过 | 非产品相关议题 |

### Subagent 分发
- Subagent 1: Track A（X个任务）
- Subagent 2: Track B（X个任务）
- Subagent 3: Track D（X个任务）
```

---

### 轨道执行优先级

当资源有限时，按以下优先级执行：

| 优先级 | 轨道 | 理由 |
|--------|------|------|
| P0 | A 公开搜索 | 基础，所有议题必选 |
| P1 | B 数据源定向 | 权威性高，数据可信 |
| P2 | D 语雀文档库 | 内部视角，效率高 |
| P3 | E/F/G 特色数据源 | 增量价值，按需激活 |
| P4 | C 专家访谈 | 成本高，需用户配合 |

---

**优势**：
- 可扩展：新数据源接入只需新增 Track
- 按需激活：三维度决策，避免无效搜索
- 并行高效：多 Subagent 同时执行，榨干每个渠道
- 权限感知：自动跳过无权限轨道

## Layer 1：假设映射与任务分解（主 Session，10-15 分钟）

### 输入
- Stage 3 输出的研究假设清单（`research_plan.md`）
- 研究议题类型（用于选择数据源组合）

### 动作

**Step 1.1：假设 → 数据点映射**

对每个假设，识别需要验证的关键数据点：

```markdown
### 假设 H1：[假设描述]

需要验证的数据点：
1. [数据点 1]：如"2024 年中国 SaaS 市场规模"
2. [数据点 2]：如"头部厂商用户渗透率"
3. [数据点 3]：如"头部竞对市场份额"
```

**Step 1.2：数据点 → 搜索任务映射**

对每个数据点，设计搜索任务：

```markdown
| 数据点 | 搜索问题 | 关键词组合 | 首选数据源 | 备选数据源 |
|-------|---------|-----------|-----------|-----------|
| 市场规模 | 2024 年中国 SaaS 市场规模？ | "SaaS 市场规模 2024 艾瑞" | 艾瑞咨询 (P1) | 头豹研究院 (P1) |
| 用户渗透率 | 头部厂商用户数及渗透率？ | "SaaS 用户数 2024" | 行业报告 (P3) | 艾瑞报告 (P1) |
```

**Step 1.3：任务分发决策**

将搜索任务分配到对应轨道：

```markdown
## 任务分发清单

### Track A（Subagent 1）：公开数据收集
- 任务 A1：[搜索问题 1]
- 任务 A2：[搜索问题 2]
- ...

### Track B（Subagent 2）：数据源定向搜索
- 任务 B1：访问 [数据源] 检索 [关键词]
- 任务 B2：访问 [数据源] 检索 [关键词]
- ...

### Track C（主 Session）：专家访谈
- [如触发，见 interview.md]
```

### 输出

1. **搜索任务清单**（分发给 Subagent）
2. **Subagent Prompt**（包含任务说明、输出格式、时间约束）

## Layer 2：并行执行（Subagent + 主 Session，30-60 分钟）

### Track A：公开数据收集（Subagent 1）

**工具**：GoogleSearch → searchJumps → InternetFetch

**执行流程**：

```
1. 接收任务清单（来自主 Session）
   ↓
2. 对每个任务：
   ├── 使用 GoogleSearch 搜索
   ├── 用 searchJumps 抓取搜索结果页面
   ├── 追溯数据原始来源（不满足于二手引用）
   └── 记录数据 + 来源 + 链接
   ↓
3. 输出结构化证据（JSON/Markdown）
   ↓
4. 回传给主 Session
```

**关键词设计公式**：

```
核心关键词 = 主体词 + 数据词 + 来源词 + 时间词

中文示例：
- "中国 SaaS 市场规模 2024 艾瑞咨询"
- "头部厂商 用户数 渗透率 2024 报告"
- "竞争分析 业务规模 券商 2024"

英文示例（Tier 2+ 默认中英双搜）：
- "China SaaS market size 2024 IDC Gartner"
- "second-hand electronics market 2024 report"
- "credit scoring industry global landscape"
```

**多语言搜索规则**：
- **Tier 2+**：默认中英双搜。每组中文关键词对应生成英文版本
- 英文搜索优先使用 GoogleSearch，覆盖海外对标企业、国际机构报告、全球趋势
- 搜索结果中的英文数据需标注来源，与中文数据交叉验证

**来源追溯规则**：

| 看到的信息 | 追溯动作 |
|-----------|---------|
| 媒体报道引用数据 | 找到原始报告/官方数据 |
| "据 XX 报告显示" | 找到 XX 报告原文 |
| "据公开数据" | 寻找具体出处 |
| 博客/自媒体分析 | 追溯数据来源，不直接引用 |

**输出格式**（Subagent 1 回传）：

```markdown
## Track A 证据清单

### 任务 A1：[搜索问题]

| 证据 ID | 数据内容 | 原始摘录 | 来源 | 发布日期 | 链接 | 初步验证 |
|--------|---------|---------|------|---------|------|---------|
| A1-01 | [提炼后的数据] | "[原文关键句]" | [来源] | [日期] | [URL] | [待验证] |
| A1-02 | [提炼后的数据] | "[原文关键句]" | [来源] | [日期] | [URL] | [待验证] |

### 任务 A2：...
```

**执行流程**：

```
   ├── 选择合适的脚本

   └── 解析 JSON 返回结果

   ├── 高互动笔记 → 重点分析
```

**脚本选择指南**：

| 场景 | 脚本 | 示例 |
|------|------|------|
| 话题搜索 | search_notes.js | `--keyword "AI编程" --sort popularity_descending` |
| 舆情扫描 | check_topics.js | `--keywords "关键词1,关键词2" --since 24h` |
| 博主动态 | fetch_user_notes.js | `--user-id "uid1,uid2" --count 5` |
| 笔记详情 | get_note.js | `--url "分享链接"` |

**搜索数量标准**：

> 搜索数量决定分析的可信度，必须遵循最低数量要求。假设每页约 10 条笔记。

| 研究阶段 | 搜索要求 | 笔记数量 | 说明 |
|---------|---------|---------|------|
| **预扫描** | 每关键词 10-20 页 | 100-200条 | 快速判断话题热度 |
| **深度研究** | 每关键词 30-50 页 | 300-500条 | 获取足够样本做定量分析 |
| **竞品分析** | 竞品品牌名搜索 15-30 页 | 150-300条 | 了解竞品声量 |

**定性定量结合要求**：

> 小红书分析必须同时输出定量指标和定性洞察，缺一不可。

| 分析类型 | 定量指标 | 定性洞察 |
|---------|---------|---------|
| **痛点分析** | 痛点占比TOP5 | 典型用户原话（3-5条） |
| **竞品分析** | 提及次数、正向比例 | 用户真实评价（好评/差评各2条） |
| **需求分析** | "求推荐"类占比 | 典型求助帖内容摘要 |

**参数组合建议**：

```bash
# 消费者舆情扫描（推荐）
node scripts/xhs/check_topics.js \
  --keywords "品牌名,产品名,竞品名" --since 24h

# 热门内容搜索
node scripts/xhs/search_notes.js \
  --keyword "关键词" --sort popularity_descending

# 最新动态追踪
node scripts/xhs/search_notes.js \
  --keyword "关键词" --sort time_descending
```

**价值判断标准**：

| 类型 | 特征 | 处理 |
|------|------|------|
| 值得关注 | 深度评测、行业趋势、KOL观点、高互动(1万+) | 重点分析 |
| 常规内容 | 转载、广告、低互动、关联度低 | 简要记录或忽略 |

```markdown
|--------|------|------|--------|---------|------|---------|
| E1-01 | [标题] | @博主 | ❤️1.2万⭐8956 | [日期] | [URL] | 高 |
| E1-02 | [标题] | @博主 | ❤️3456⭐2100 | [日期] | [URL] | 中 |

- [观点1]
- [观点2]
```

**定量分析方法论**：

> 小红书数据不仅要定性分析，更要定量量化，形成可比较、可追溯的数据洞察。

| 维度 | 指标 | 计算方式 | 价值 |
|------|------|----------|------|
| **热度分析** | 关键词热度指数 | 笔记数 × 平均互动量 / 1000 | 判断话题关注度 |
| **情感分析** | 正/负向比例 | 正向笔记数 / 总笔记数 | 判断舆情倾向 |
| **痛点聚类** | 痛点TOP5 | 高频词 + 互动量加权 | 识别核心需求 |
| **竞品提及** | 品牌提及频次 | 笔记中品牌名出现次数 | 竞品声量对比 |
| **需求洞察** | 未满足需求 | "求推荐"类帖子占比 | 发现机会点 |

**定量分析输出格式**：

```markdown

| 品牌 | 提及次数 | 正向比例 |
|------|---------|---------|
| 品牌1 | X | X% |

- 核心未满足需求：[需求描述]
```

## 常见错误与规避

✅ 使用统一的输出模板，主 Session 负责标准化

### 错误 3：三角验证流于形式

❌ 用同一机构的多份报告当作多个独立来源

### 错误 4：进度不透明

✅ 每个节点广播进度和核心发现

## 附录：完整示例

### 示例：企业服务 SaaS 行业研究

**假设 H1**：中国企业服务 SaaS 市场处于快速增长期，头部厂商面临新进入者威胁

**Layer 1 任务分解**：

| 数据点 | 搜索问题 | 关键词 | 轨道 | 目标数据源 |
|-------|---------|-------|------|-----------|
| 市场规模 | 2024 年中国 SaaS 市场规模？ | "SaaS 市场规模 2024" | A | 艾瑞、头豹 |
| 增长率 | 近 5 年市场增长率？ | "SaaS CAGR 2020-2024" | B | 艾瑞官网 |
| 竞对份额 | 头部厂商市场份额？ | "SaaS 厂商 份额 2024" | A | 券商研报 |
| 渗透率 | SaaS 服务企业渗透率？ | "SaaS 渗透率 2024" | B | CNNIC、易观 |

**Layer 2 执行**：
- Subagent 1（Track A）：执行 4 个 Google 搜索任务
- Subagent 2（Track B）：访问艾瑞、CNNIC 官网检索

**Layer 3 整合**：
- 汇总 8 个证据
- 执行三角验证：市场规模（B 级，2 源验证）、增长率（C 级，单一来源）
- 输出数据验证清单

## Stage 4 完成后：框架分析结论（必须）

> Stage 4 所有 Track 完成后，**必须**产出独立的框架分析结论章节。
> 框架分析结论不是散落在搜索结果里的零碎信息，而是对 Stage 2 选定的每个框架的系统性应用总结。

**执行要求**：

对 Stage 2 确认的每个框架（主框架 + 增强框架），在 `evidence_base.md` 末尾产出独立结论段落：

```markdown
## 框架分析结论

### [主框架名称] 分析结论
- **核心发现**：[基于框架维度的 3-5 条关键发现]
- **数据支撑**：[引用 evidence_base 中的证据 ID]
- **初步判断**：[框架视角下的整体判断]

### [增强框架 1] 分析结论
- **核心发现**：[2-3 条关键发现]
- **与主框架的关系**：[如何深化主框架某个步骤的分析]

### [增强框架 2] 分析结论
...
```

**目的**：确保框架不只是「选了」，而是「用了」，且分析结论独立可见、可追溯。
